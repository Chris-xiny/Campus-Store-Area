## 逐项详情

### 1. 新增 AMQP 依赖

**做了什么：** 在 `pom.xml` 的 `<dependencies>` 段添加 `spring-boot-starter-amqp`。

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-amqp</artifactId>
</dependency>
```

**为什么这么操作：** Spring AMQP 是 Spring 生态对接 RabbitMQ 的标准 Starter，提供 `RabbitTemplate`（生产者）、`@RabbitListener`（消费者）、`SimpleMessageListenerContainer`（容器）等核心组件。引入后 Spring Boot 自动配置 `ConnectionFactory`、`RabbitAdmin` 等 Bean，开箱即用。

**面试可聊增量：**
> "Spring Boot 引入 AMQP Starter 后会自动装配 RabbitAutoConfiguration，提供 RabbitTemplate 和 SimpleRabbitListenerContainerFactory。生产者用 RabbitTemplate 发消息，消费者用 @RabbitListener 注解声明监听队列。"

可延伸：`spring-boot-starter-amqp` 自动装配原理、`ConnectionFactory` 配置、AMQP 协议与 JMS 的区别。

### 2. RabbitMQ 连接与消费者配置

**做了什么：** 在 `application.yaml` 新增 RabbitMQ 配置段：

```yaml
spring:
  rabbitmq:
    host: 192.168.100.128
    port: 5672
    username: guest
    password: guest
    virtual-host: /
    publisher-confirm-type: correlated
    publisher-returns: true
    listener:
      simple:
        acknowledge-mode: manual
        prefetch: 20
        concurrency: 5
        max-concurrency: 10
        retry:
          enabled: true
          initial-interval: 1000
          multiplier: 2
          max-attempts: 3
```

**为什么这么操作：** 逐项解释每个配置的设计意图：

| 配置项 | 值 | 设计意图 |
|:---|:---|:---|
| `publisher-confirm-type` | `correlated` | 异步回调确认，Broker 收到消息后触发 ConfirmCallback，不阻塞主线程 |
| `publisher-returns` | `true` | 开启 Returns 回调，消息无法路由到队列时触发 ReturnsCallback，防止消息丢失 |
| `acknowledge-mode` | `manual` | 手动 ACK，消费者处理完业务逻辑后才发 basicAck，避免 Broker 在投递后就认为消费成功 |
| `prefetch` | `20` | 每个消费线程一次从 Broker 缓存 20 条消息，减少网络往返，提升吞吐 |
| `concurrency` | `5` | 初始 5 个消费线程 |
| `max-concurrency` | `10` | 消息堆积时自动扩到 10 个线程，弹性伸缩 |
| `retry` | enabled | 消费失败自动重试 3 次（1s → 2s → 4s），瞬时故障可自愈 |

**面试可聊增量：**
> "消费者我用的是手动 ACK 模式。prefetch=20 让每个消费线程从 Broker 预拉 20 条消息缓存在本地，处理完一条立刻拿下一条，不用每次网络往返。concurrency=5-10 意味着初始 5 个消费线程，消息堆积时自动扩到 10 个，相当于弹性线程池。"

可延伸：`prefetch` 对内存的影响、`concurrency` 与 `@RabbitListener` 注解的优先级、手动 ACK 的 basicNack + requeue 策略。

### 3. 定义 MQ 常量

**做了什么：** 新建 `mq/RabbitMqConstants.java`，集中管理队列、交换机、路由键三个常量：

```java
public class RabbitMqConstants {
    public static final String SECKILL_ORDER_QUEUE = "seckill.order.queue";
    public static final String SECKILL_ORDER_EXCHANGE = "seckill.order.exchange";
    public static final String SECKILL_ORDER_ROUTING_KEY = "seckill.order";
```

**面试可聊增量：**

可延伸：Direct Exchange 的路由规则、routing key 与 binding key 的精确匹配、四种 Exchange 类型对比。

### 4. RabbitMQ 基础配置

**做了什么：** 新建 `config/RabbitMQConfig.java`，声明三件事：

1. **消息转换器**：`Jackson2JsonMessageConverter` 替换默认的 JDK 序列化
2. **队列声明**：`QueueBuilder.durable(...)` 创建持久化队列
3. **交换机 + 绑定**：`DirectExchange` + `BindingBuilder.bind(queue).to(exchange).with(routingKey)`

**为什么这么操作：**

- **Jackson2JsonMessageConverter**：默认 JDK 序列化后的消息是二进制乱码，在 RabbitMQ 管理界面和日志里不可读，而且 Java 以外的语言无法消费。Jackson JSON 序列化后消息体是标准 JSON，跨语言可读，调试时直接看管理界面就能确认消息内容。

- **持久化队列**（`durable=true`）：RabbitMQ 重启后队列和消息不丢失。配合消息的 `deliveryMode=2`（AMQP 默认），消息写入磁盘，即使 Broker 宕机数据也在。

- **Direct Exchange**：秒杀场景只需要把消息精确投递到一个队列，Direct Exchange 按 routing key 精确匹配，是最简单也最高效的选择。Topic/Fanout 在这个场景下是过度设计。

**面试可聊增量：**
> "消息序列化我用了 Jackson2JsonMessageConverter 替代默认的 JDK 序列化，好处是消息体是标准 JSON，RabbitMQ 管理界面可直接查看，也支持跨语言消费。队列声明为 durable，配合消息默认的 deliveryMode=2（持久化），Broker 重启后消息不丢。"

可延伸：四种 Exchange 类型（Direct/Topic/Fanout/Headers）的适用场景、消息持久化的性能开销、惰性队列（Lazy Queue）。

### 5. RabbitTemplate 配置 Publisher Confirm + Returns 回调

**做了什么：** 在 `RabbitMQConfig.java` 中配置 `RabbitTemplate` Bean，设置两个回调：

```java
rabbitTemplate.setConfirmCallback((correlationData, ack, cause) -> {
    if (!ack) {
        log.error("消息未到达 Broker, correlationId={}, cause={}",
                correlationData.getId(), cause);
    }
});
rabbitTemplate.setReturnsCallback(returned -> {
    log.error("消息未路由到队列, exchange={}, routingKey={}, replyCode={}, replyText={}",
            returned.getExchange(), returned.getRoutingKey(),
            returned.getReplyCode(), returned.getReplyText());
});
```

**为什么这么操作：** 消息从 Producer 到 Queue 要经过两跳：

```
Producer → [第一跳] → Exchange → [第二跳] → Queue
```

- **ConfirmCallback** 保证第一跳：Broker 收到了消息。`ack=true` 表示 Broker 已接收，`ack=false` 表示 Broker 拒绝或丢失（如交换机不存在、连接中断）。
- **ReturnsCallback** 保证第二跳：Exchange 成功把消息路由到了某个 Queue。如果 routing key 匹配不上任何队列（比如队列被误删），触发 Returns 回调。

两个回调组合起来 = **消息从 Producer 到 Queue 的全链路可靠投递**。

**面试可聊增量：**
> "生产者可靠性我做了两层保障：publisher-confirm-type=correlated 开启异步确认回调，Broker 收到消息后触发 ConfirmCallback；publisher-returns=true 开启 Returns 回调，消息无法路由到队列时也能感知。两者组合保证了从 Producer 到 Queue 的全链路不丢消息。"

可延伸：`correlated` vs `simple` 确认模式的区别、ConfirmCallback 中的 correlationId 设计、生产环境 Confirm 失败的重试策略。

### 6. 生产者：OrderCorrelationData + send()

**做了什么：** 新建 `mq/VoucherOrderProducer.java`，核心是 `send()` 方法：

```java
public void send(VoucherOrder order) {
    OrderCorrelationData correlationData = new OrderCorrelationData();
    correlationData.setId(order.getId().toString());
    correlationData.setOrderId(order.getId());
    correlationData.setVoucherId(order.getVoucherId());
    correlationData.setUserId(order.getUserId());

    rabbitTemplate.convertAndSend(
            RabbitMqConstants.SECKILL_ORDER_EXCHANGE,
            RabbitMqConstants.SECKILL_ORDER_ROUTING_KEY,
            order,
            correlationData
    );
}
```

其中 `OrderCorrelationData` 是自定义静态内部类，继承 `CorrelationData`，额外携带 `orderId`、`voucherId`、`userId` 字段。

**为什么这么操作：**

Spring Boot 2.7 使用的 Spring AMQP 2.4.x 版本，`CorrelationData` 类没有 `getMetadata()` 方法（这是 3.0+ 才加的 API）。为了在 ConfirmCallback 回调中拿到业务字段（用于失败时回滚 Redis 预扣），只能自定义子类扩展。

这是版本兼容性问题的实际解决案例，面试时能讲出来会加分。

**面试可聊增量：**
> "CorrelationData 在 Spring AMQP 2.4.x 版本里只有 id 字段，不方便在回调里拿业务信息。我自定义了 OrderCorrelationData 子类扩展了 orderId、voucherId、userId 字段，这样 Confirm 失败时可以直接从 correlationData 拿到完整上下文做补偿。"

可延伸：CorrelationData 的 id 全局唯一性设计（Redis ID Worker）、Confirm 回调中的补偿策略、事务消息（RocketMQ half message）对比。

### 7. Lua 脚本精简：移除 XADD

**做了什么：** 重写 `seckill.lua`，移除 `XADD` 命令和 `orderId` 参数：

```lua
-- 改造前（Redis Stream 方案）
local orderId = ARGV[3]
redis.call('xadd', 'stream.voucher.order', '*', 'userId', userId, 'voucherId', voucherId, 'orderId', orderId)

-- 改造后（RabbitMQ 方案）
-- 不需要 XADD，不需要 orderId 参数
-- 只返回 0（成功）/ 1（库存不足）/ 2（重复购买）
```

**为什么这么操作：** Redis Stream 时代，Lua 脚本需要把订单信息写入 Stream（XADD），所以需要传入 orderId。迁移到 RabbitMQ 后，消息投递由 Java 层的 `producer.send()` 负责，Lua 脚本的职责缩小为纯粹的"Redis 原子预扣"——判库存、判一人一单、扣库存、记购买标记，返回状态码即可。职责更单一，脚本更轻量。

**面试可聊增量：**
> "Lua 脚本改造后职责更纯粹了，只做 Redis 层面的原子预扣（判库存 + 判一人一单 + 扣减 + 打标），返回状态码 0/1/2。消息投递的职责上移到 Java 层的 RabbitMQ Producer，Lua 和 MQ 各司其职。"

可延伸：Lua 脚本的原子性保证原理（Redis 单线程执行）、EVALSHA 优化、Redis 7 Function 替代 Lua。

### 8. 消费者：手动 ACK + 幂等校验

**做了什么：** 新建 `mq/VoucherOrderConsumer.java`，核心方法：

```java
@RabbitListener(queues = SECKILL_ORDER_QUEUE, concurrency = "5-10")
public void onMessage(VoucherOrder order, Channel channel,
                      @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag,
                      @Header(AmqpHeaders.REDELIVERED) boolean redelivered) {
    try {
        voucherOrderService.handleVoucherOrder(order);  // 幂等 + 落库
        channel.basicAck(deliveryTag, false);            // 成功：ACK
    } catch (Exception e) {
        channel.basicNack(deliveryTag, false, false);   // 失败：NACK + 不重入队
    }
}
```

**为什么这么操作：** 三个设计决策：

1. **手动 ACK**：自动 ACK 模式下，Broker 把消息投递给消费者就认为"消费成功"，如果消费者处理到一半挂了，消息就丢了。手动 ACK 保证"业务处理完才确认"，是消息可靠消费的基石。

2. **幂等校验**：`handleVoucherOrder()` 内部先查 `tb_voucher_order` 是否已存在该订单 ID，存在则直接返回。应对网络抖动导致的消息重发（Broker 没收到 ACK 会重发），防止重复落库。

3. **`basicNack(requeue=false)`**：消费失败时不重新入队，避免同一条"毒消息"（数据有误导致的永久失败）反复消费形成死循环。生产环境应该发到死信队列（DLX），由人工或补偿任务处理。

**面试可聊增量：**
> "消费者我用了手动 ACK 模式。basicAck 在业务处理成功后调用，basicNack 在异常时调用且 requeue=false，避免毒消息死循环。幂等性通过订单 ID 查询实现——先查 DB 是否已存在，已存在则跳过，保证重复消息不会重复落库。"

可延伸：手动 ACK vs 自动 ACK 的吞吐差异、`basicNack(multiple=true)` 批量确认、死信队列（DLX）设计、消息 TTL + 死信实现延迟队列。

### 9. Service 层重构：移除 Stream，改用 MQ 投递

**做了什么：** 重写 `VoucherOrderServiceImpl.seckillVoucher()` 方法：

```
改造前：Lua 执行 → XADD 写入 Stream → 返回
改造后：Lua 执行 → 生成 orderId → producer.send() 投递 MQ → 返回
```

新增 `rollbackRedisPreDeduct()` 方法：MQ 发送失败时回滚 Redis 中已扣减的库存和购买标记。

移除 `@PostConstruct init()` 方法和所有 Stream 消费逻辑（`handlePendingList`、`doSeckill`）。

**为什么这么操作：**

- **移除 `@PostConstruct init()`**：Stream 时代需要在 Bean 初始化后启动一个 `while(true)` 的死循环线程来消费 Stream，这个线程是单线程模型，且 Bean 初始化阶段启动不利于测试和资源管理。RabbitMQ 的 `@RabbitListener` 由 Spring 容器管理生命周期，自动启停，更优雅。

- **新增 `rollbackRedisPreDeduct()`**：Lua 脚本已经在 Redis 中扣了库存和购买标记，但 MQ 发送可能失败（网络抖动、Broker 宕机）。如果不回滚，就出现"Redis 扣了但 DB 没扣"的不一致。回滚操作是 `increment` 库存 + `remove` 购买标记，都是原子操作。

**面试可聊增量：**
> "seckillVoucher 的流程是：Lua 原子预扣 → 生成订单号 → 投递 RabbitMQ → 立即返回。如果 MQ 发送失败，我会回滚 Redis 中已扣减的库存和购买标记，保证 Redis 和 DB 的一致性。回滚用的 increment 和 SREM 都是 Redis 原子操作。"

可延伸：分布式事务（Seata AT/TCC）、最终一致性 vs 强一致性、Redis Lua 回滚的局限性（非事务性）。

### 11. 删除废弃线程池 Bean

**做了什么：** 从 `AsyncConfig.java` 中删除 `voucherOrderExecutor` Bean（核心 10 / 最大 20 / 队列 500）。

**为什么这么操作：** 这个线程池原来给 `@Async("voucherOrderExecutor")` 注解用，驱动 Stream 消费线程。迁移到 RabbitMQ 后：

- 消费者线程由 `@RabbitListener(concurrency="5-10")` + `SimpleMessageListenerContainer` 管理
- `voucherOrderExecutor` 在整个项目中没有任何引用
- `AsyncTaskUtils` 里唯一保留的 `rebuildCacheAsync()` 用的是 `cacheExecutor`

留着死 Bean 面试会被追问"这个池在哪用的"，答不上来就减分。

**面试可聊增量：**
> "RabbitMQ 的 @RabbitListener 背后是 SimpleMessageListenerContainer，它自己维护消费线程池，支持 concurrency 弹性伸缩。之前手动创建的 ThreadPoolTaskExecutor 已经不需要了，我做了清理。"

可延伸：`SimpleMessageListenerContainer` vs `DirectMessageListenerContainer`、`concurrency` 参数对消费吞吐的影响、`prefetch` 与 `concurrency` 的调优配合。

### 数据分析

**瓶颈不在消费者侧。** 200 个请求在 Lua 预扣阶段全部成功（库存 201 充足），进入 MQ 的也只有 200 条消息。`concurrency=5-10` 消费 200 条绑绑有余，DB 落库不是瓶颈。

**101ms 的构成：**
- HTTP 连接建立 + Tomcat 线程分配：~30ms（200 个并发连接）
- Redis Lua 脚本执行：~5ms
- RedisIdWorker 生成订单号：~5ms
- RabbitMQ Producer send + Publisher Confirm：~30ms（跨机器网络往返）
- Spring Boot Controller 序列化 + 响应返回：~10ms
- 其他开销（MDC、日志、拦截器链）：~20ms

**跟之前 1000 线程 440ms 的对比：** 那次 80% 错误率是因为 1000 线程共享 200 个 token 导致"重复购买"拦截，440ms 是 Tomcat 线程池（默认 200 线程）排队等待导致的，和 MQ 性能无关。同等 200 线程条件下，RabbitMQ 方案的 101ms 已经是非常健康的数据。

**面试可聊增量：**

## RabbitMQ 安装踩坑记录

在虚拟机上用 Docker 安装 RabbitMQ 时遇到 `Too short cookie string` 启动失败。原因是 `rabbitmq:3-management`（latest tag → 4.x 版本）对 Erlang Cookie 有更严格的校验要求。

**解决方案：** 指定 `rabbitmq:3.13-management` 稳定版 + 20 字符以上的 cookie：

```bash
docker run -d --name rabbitmq \
  -p 5672:5672 \
  -p 15672:15672 \
  -e RABBITMQ_ERLANG_COOKIE=SWQOKODSQALRPCLNMEQGAA \
  rabbitmq:3.13-management
```

**教训：** Docker 镜像的 `latest` tag 不保证向后兼容，生产环境应锁定具体版本号。

## 核心原理速查（面试用）

### RabbitMQ 消息投递全链路

```
Producer → ConfirmCallback ← Exchange → ReturnsCallback ← Queue → Consumer → ACK/NACK
         |______ 第一跳 ______|        |______ 第二跳 ______|    |______ 第三跳 ______|
```

### 可靠性三板斧

| 环节 | 机制 | 应对的故障 |
|:---|:---|:---|
| Producer → Broker | Publisher Confirm + Returns | 网络中断、交换机/队列不存在 |
| Broker 存储 | 消息持久化（durable queue + deliveryMode=2） | Broker 宕机、磁盘故障 |
| Broker → Consumer | 手动 ACK + 幂等校验 | 消费者处理失败、消息重发 |

### AMQP 四种 Exchange 对比

| 类型 | 路由规则 | 本项目使用 |
|:---|:---|:---|
| Direct | routing key 精确匹配 | **是** — 秒杀消息只需精确投递到一个队列 |
| Topic | routing key 通配符匹配（`*` / `#`） | 否 — 场景不需要多级路由 |
| Fanout | 广播到所有绑定队列 | 否 — 秒杀只需一个队列 |
| Headers | 消息 header 属性匹配 | 否 — 性能差，少用 |
