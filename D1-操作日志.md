## 逐项详情

### 1. 新增参数校验依赖

**做了什么：** 在 `pom.xml` 的 `<dependencies>` 段添加 `spring-boot-starter-validation`。

**为什么这么操作：** 原项目完全没有使用 Bean Validation（整个项目找不到一个 `@Valid`、`@NotBlank`、`@Pattern`），所有 Controller 的参数校验都靠手写 `if` 判断。这不符合企业级工程规范，也让后续的"参数校验异常处理"无从谈起。引入 starter 后可以用注解声明校验规则，并由框架自动触发 `MethodArgumentNotValidException`。

**面试可聊增量：**
> "我引入了 JSR-303 参数校验，把手机号正则、验证码格式等校验从业务代码下沉到 DTO 注解上，配合全局异常处理器返回字段级的错误信息，避免脏数据进入 Service 层。"

可延伸：`@Valid` 和 `@Validated` 的区别、分组校验、自定义校验注解。

### 2. Redisson 配置外置化

**做了什么：**
- `application.yaml` 新增独立配置段：
  ```yaml
  redisson:
    address: redis://192.168.100.128:6379
    password: '000000'
    database: 0
  ```
- `RedissonConfig.java` 删除 36 行硬编码（含 2 个被注释的 `redissonClient2/3` Bean），改用 `@Value` 从 yaml 注入。

**为什么这么操作：** 原代码把 `redis://192.168.100.128:6379` 和密码 `000000` 硬编码在 Java 类里，且与 `application.yaml` 的 `spring.redis.*` 配置重复维护。一旦切换环境（开发/测试/生产）就要改代码重新编译。外置化后可通过 profile 或环境变量覆盖，符合 12-Factor 配置规范。

**面试可聊增量：**
> "我把 Redisson 配置从 Java 硬编码外置到 application.yaml，通过 @Value 注入，避免配置与代码耦合，方便不同环境切换。"

可延伸：`@ConfigurationProperties` vs `@Value`、Spring Profile、配置加密（jasypt）。

### 3. 统一响应体加 traceId + timestamp

**做了什么：**
- `Result.java` 新增两个字段：`traceId`（String）+ `timestamp`（Long）。
- 把所有静态工厂方法（`ok`/`fail`）重构为统一的 `build()` 私有方法，在构建时自动从 MDC 取 `traceId`、填充当前时间戳。

**为什么这么操作：** 原 `Result` 只有 `success/errorMsg/data/total` 四个字段，出问题后无法定位是哪一次请求的报错。加上 traceId 后：
1. 前端可以直接把 traceId 展示给用户，客服反馈时后端能秒级定位日志；
2. timestamp 让前端知道响应生成的时间点，便于缓存策略和超时判断；
3. 为后续接入链路追踪（SkyWalking、Zipkin）预留扩展点。

**面试可聊增量：**
> "我在统一响应体里加了 traceId 和 timestamp 两个字段。traceId 通过 MDC 从拦截器透传过来，每次响应都带上，方便线上问题定位。"

可延伸：MDC 原理（基于 ThreadLocal）、子线程传递 MDC、响应式场景下的 traceId 传播。

### 4. 新建业务异常类

**做了什么：** 新建 `BusinessException.java`（位于 `com.hmdp.config` 包），继承 `RuntimeException`，带 `code` 字段和两个构造方法。

**为什么这么操作：** 原项目所有错误都靠 `Result.fail("xxx")` 返回，Service 层为了中断流程会写一堆 `if (count > 0) return Result.fail(...)`。这种写法有两个问题：
1. Service 返回 `Result` 违反了分层原则（Service 应返回领域对象，由 Controller 包装）；
2. 嵌套调用时错误信息无法穿透，必须层层判断返回值。

引入 `BusinessException` 后，Service 层直接 `throw new BusinessException("库存不足")`，由全局异常处理器统一拦截，Service 返回类型可以干净地改成领域对象。

**面试可聊增量：**
> "我定义了 BusinessException 业务异常类，把可预期的业务失败（如库存不足、未登录）和系统级 RuntimeException 区分开。业务异常走 warn 日志不打堆栈，系统异常才打 error 堆栈，这样日志更干净。"

可延伸：checked vs unchecked exception、`@ExceptionHandler` 优先级、跨模块异常传递。

---

### 5. 全局异常处理三分

**做了什么：** 重写 `WebExceptionAdvice.java`，新增三类处理器：
1. `@ExceptionHandler(BusinessException.class)`：业务异常，warn 日志
2. `@ExceptionHandler(MethodArgumentNotValidException.class)`：`@RequestBody` + `@Valid` 触发的校验失败
3. `@ExceptionHandler(BindException.class)`：表单绑定校验失败
4. `@ExceptionHandler(RuntimeException.class)`：兜底系统异常，error 日志带 traceId

**为什么这么操作：** 原 `WebExceptionAdvice` 只有一个 `RuntimeException` 处理器，返回硬编码的 "服务器异常"。这导致：
- 业务错误（如 "库存不足"）被吞掉，前端只能看到笼统错误；
- 参数校验失败时框架直接抛 400，前端拿不到字段级错误信息；
- 所有异常都打 error 堆栈，日志噪音大。

改造后每种异常走不同处理路径，错误信息对前端更友好，日志分级更合理。

**面试可聊增量：**
> "全局异常处理我分了三层：业务异常走 warn、参数校验失败返回字段级错误信息（如 phone:手机号格式错误）、系统异常才打 error 堆栈。这样既能让前端拿到精确错误，又不让业务失败污染 error 日志。"

可延伸：`@RestControllerAdvice` vs `@ControllerAdvice`、`HandlerExceptionResolver`、异常处理链顺序。

---

### 6. 新建链路追踪拦截器

**做了什么：** 新建 `TraceIdInterceptor.java`（位于 `com.hmdp.interceptor` 包），在 `preHandle` 时：
1. 从请求头 `X-Trace-Id` 取 traceId（支持前端透传）
2. 取不到则用 Hutool 的 `UUID.fastUUID()` 生成
3. 写入 MDC（供日志和 Result 使用）
4. 回写到响应头（前端可拿到）

`afterCompletion` 时清理 MDC 防止线程池污染。

**为什么这么操作：** Spring Boot 默认日志没有请求级标识，并发场景下多个请求的日志交织在一起难以追踪。MDC + 拦截器是最轻量的链路追踪方案，无需引入 SkyWalking 这种重型中间件，对实习生项目正合适。

**面试可聊增量：**
> "我做了一个 TraceIdInterceptor，每个请求自动分配一个 traceId。如果前端传 X-Trace-Id 请求头就透传，否则服务端生成。traceId 写入 MDC 后，日志和响应体都能自动带上。"

可延伸：MDC 的 ThreadLocal 本质、线程池场景下的 MDC 丢失问题（TaskDecorator 解决）、OpenTelemetry 标准。

### 9. 统一 Token TTL 单位为天

**做了什么：**
- `UserServiceImpl.login()` 中 `TimeUnit.MINUTES` → `TimeUnit.DAYS`
- `RefreshTokenInterceptor` 保持不变（已经是 `TimeUnit.DAYS`）
- 加注释 `//5.4 设置Token有效期（单位：天）`

`LOGIN_USER_TTL = 30L`，统一后所有使用处都表示"30 天"。

**为什么这么操作：** 这是排查出来的**真实 bug**：登录时设置 TTL 为 30 分钟，但只要用户 30 分钟内有一次请求，刷新拦截器就会把 TTL 重置为 30 天。结果就是用户登录 30 分钟后可能突然下线，也可能一直在线，行为不可预测。统一为天后，行为可预期、用户体验一致（类似微信登录 30 天免登录）。

**面试可聊增量：**
> "排查代码时发现一个 bug：登录时 Token 有效期设的是分钟，刷新时用的是天，导致 TTL 语义混乱。我统一改为天，让登录态行为可预期。"

可延伸：JWT vs Redis Token 的过期策略、滑动窗口过期、Token 续期与强制下线。

### 10. 拦截器迁移到 interceptor 包

**做了什么：**
- 新建 `com.hmdp.interceptor` 包
- 把 `LoginInterceptor`、`RefreshTokenInterceptor` 从 `utils` 包移动到 `interceptor` 包
- 删除原 `utils/LoginInterceptor.java` 和 `utils/RefreshTokenInterceptor.java`
- 同步更新所有引用（`MVCConfig.java` 等）

**为什么这么操作：** `utils` 包按定义应该放"无状态的工具类"（如 `UserHolder`、`RedisIdWorker`），而 Interceptor 是 Spring MVC 的扩展点，属于 web 层组件，应该独立成包。原项目把拦截器塞进 `utils` 是典型的分层混乱，面试官一眼就能看出不合理。

**面试可聊增量：**
> "我把项目包结构整理了一下：拦截器从 utils 包移到独立的 interceptor 包，utils 只保留无状态工具类。这样包结构更符合分层语义。"

可延伸：DDD 分层架构（interfaces/application/domain/infrastructure）、Controller-Service-Mapper 分层边界。

### 11. 拦截器链编排

**做了什么：** 在 `MVCConfig.java` 中注册三个拦截器并指定 order：
- `TraceIdInterceptor`：`addPathPatterns("/**")`，`order(-2)`
- `RefreshTokenInterceptor`：`addPathPatterns("/**")`，`order(0)`
- `LoginInterceptor`：排除公开路径，`order(1)`

**为什么这么操作：** order 数字越小越先执行。链路追踪必须最先执行（保证后续拦截器和业务都能拿到 traceId），刷新 Token 其次（先解析出用户），登录拦截最后（此时 UserHolder 才有值可以判断）。原来的注册顺序虽然也是 RefreshToken 在前，但没有显式 order 控制，一旦新增拦截器容易乱。

**面试可聊增量：**
> "三个拦截器用 order 显式控制顺序：TraceId(-2) → RefreshToken(0) → Login(1)。TraceId 必须最先执行，否则后续拦截器的日志就追踪不到。"

可延伸：拦截器 vs Filter vs AOP 的执行顺序、Spring Security 的 Filter Chain、异常在拦截器链中的传播。

### 12. 登录 DTO 参数校验

**做了什么：**
- `LoginFormDTO` 加上校验注解：
  ```java
  @NotBlank(message = "手机号不能为空")
  @Pattern(regexp = "^1[3-9]\\d{9}$", message = "手机号格式错误")
  private String phone;

  @NotBlank(message = "验证码不能为空")
  @Pattern(regexp = "^\\d{4,6}$", message = "验证码格式错误")
  private String code;
  ```
- `UserController.login()` 加上 `@Valid` 触发校验

**为什么这么操作：** 原 `UserServiceImpl.login()` 里手写 `if (RegexUtils.isPhoneInvalid(phone))` 做校验，校验失败时返回 `Result.fail(...)`。这种写法问题：
1. Service 层承担了本该在 Controller 层做的入参校验；
2. 错误信息是固定字符串，前端不知道哪个字段错；
3. 每个 Controller 都要重复写校验逻辑。

改造后校验规则声明在 DTO 上，框架自动触发，错误信息自动带字段名。

**面试可聊增量：**
> "我把手机号和验证码的校验规则用 JSR-303 注解声明在 DTO 上，Controller 用 @Valid 触发，校验失败由全局异常处理器返回字段级错误（如 phone:手机号格式错误），Service 层不再关心入参格式。"

可延伸：正则表达式原理、分组校验（`@Validated(CreateGroup.class)`）、自定义 `ConstraintValidator`。

### 13. 登出接口补全

**做了什么：** 重写 `UserController.logout()`：
```java
@PostMapping("/logout")
public Result logout(HttpServletRequest request){
    String token = request.getHeader("authorization");
    if (token != null && !token.isEmpty()) {
        stringRedisTemplate.delete(RedisConstants.LOGIN_USER_KEY + token);
    }
    UserHolder.removeUser();
    return Result.ok();
}
```

**为什么这么操作：** 原代码直接 `return Result.fail("功能未完成")`——这是课程留下的空壳。登录/登出是鉴权闭环的基本操作，缺少登出意味着用户无法主动结束会话、无法切换账号、Token 只能等过期。作为全栈项目，这是必须补齐的最小功能。

**面试可聊增量：**
> "登录登出我做了完整闭环：登出时从请求头取 token、删除 Redis 中的用户信息、清除 ThreadLocal 中的 UserHolder，避免线程池复用导致的上下文污染。"

可延伸：`UserHolder.removeUser()` 必须在 `afterCompletion` 兜底、线程池场景下的 ThreadLocal 泄漏、单点登录的登出传播。
