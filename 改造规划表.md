## 一、项目现状与改造思路

**当前项目已具备的能力（保留复用，不重写）：**
- 秒杀模块：Lua脚本 + Redis Stream + 阻塞队列三版实现，含Redisson分布式锁
- 缓存一致性：CacheClient工具类，封装互斥锁和逻辑过期两套方案
- Redis ID生成器、登录拦截器链、异步线程池、统一响应Result
- 商户查询（含缓存）、优惠券管理、登录注册（登出为空壳）

**当前项目的短板（本次改造重点）：**
- 5个后端功能空壳：博客评论、关注粉丝、用户签到、附近商户GEO、Feed流
- 博客点赞仍为直接SQL操作，未改Redis SET方案
- 完全没有前端代码、没有Docker、没有Swagger文档、没有参数校验
- 大量注释掉的旧代码、硬编码配置、命名拼写错误

**改造原则：不重写已有优质代码，只做"增量"——补全空壳、加前端、加工程化、加亮点。**

## 四、简历项目概述话术（3段，可直接套用）

**话术一（技术决策型，推荐）：**

> 基于Spring Boot + Vue 3 + Redis的本地生活服务平台，涵盖商户查询、优惠券秒杀、探店社区、用户签到、关注互动等核心业务。后端秒杀模块我做了两版方案对比：课程原版的Lua脚本+阻塞队列，和我自己改的Redisson分布式锁+RabbitMQ，经JMeter压测后者QPS提升约6倍，最终采用Redisson方案支持可重入和自动续期。缓存层按场景分离策略：热点Key用互斥锁保证一致性，高并发读场景用逻辑过期换取吞吐。前端独立搭建Vue 3 + TypeScript工程，封装统一请求层和路由守卫，开发签到日历、博客Feed流等交互组件，使用WebSocket实现评论与点赞的实时通知。项目使用Docker Compose一键编排部署，集成Knife4j接口文档。

**话术二（问题导向型）：**

> 个人独立开发的全栈项目，在原有教学项目基础上完成了大量增量开发。针对课程未实现的用户签到、关注互动、附近商户等5个空模块，我基于Redis BitMap/SET/GEO分别设计了对应方案，并实现了Feed流的推拉结合策略（普通用户推模式+大V拉模式）。针对课程秒杀方案的可维护性问题，用Redisson替换Lua脚本，并通过JMeter压测验证性能。前端从零搭建Vue 3工程，开发签到日历组件和WebSocket消息中心，完成从接口设计到页面交互的全链路开发。

**话术三（成长叙事型）：**

> 为系统提升全栈开发能力，我基于一个本地生活服务平台的教学项目进行深度扩展。后端补全了签到、关注、评论、GEO、Feed流等核心业务模块，用AOP实现操作日志的无侵入记录，用Knife4j规范接口文档；前端独立搭建Vue 3 + TypeScript工程，开发了商户列表、签到日历、博客瀑布流等页面，并通过WebSocket实现实时消息推送。项目使用Docker Compose编排MySQL、Redis、Nginx和前后端服务，编写了完整的README并录制了演示视频。整个改造周期2周，代码增量约6000行。

## 五、简历避坑建议（2条，针对本项目）

**避坑一：简历里绝对不能出现课程原话术，必须出现"课程没讲的东西"。**

普通黑马点评简历的高频雷区："使用Redis的Lua脚本解决超卖问题"、"一人一单使用分布式锁"、"使用Redis Stream实现异步秒杀"——这三句话面试官一天能看二十遍。你的简历必须换成：
- "JMeter压测对比Lua脚本与Redisson两种锁方案，最终选用Redisson，QPS提升约6倍"
- "按读写场景分离缓存策略：热点Key互斥锁、高并发读逻辑过期"
- "补全课程未实现的签到/关注/评论/GEO/Feed流5个模块"
- "前端独立搭建Vue 3工程，开发签到日历可交互组件"

**这些关键词课程里根本没有**，面试官一看就知道你不是跟课敲的。

**避坑二：准备好"为什么改"的说辞，不要只说"做了什么"。**

面试官追问秒杀方案时，最差的回答是复述课程原话（"用Lua脚本预扣库存……"）。建议用三段式回答：**课程方案是X → 我压测/思考后发现Y问题 → 所以改为Z方案**。举例：

> "课程秒杀方案是Lua脚本+阻塞队列，我做完之后用JMeter压测发现两个问题：一是Lua脚本的可维护性比较差，业务逻辑改动需要改脚本；二是同步落库在并发上千时成为瓶颈。所以我把锁换成Redisson支持可重入，把异步落库换成RabbitMQ，测下来QPS从280提升到1800左右。不过MQ引入了消息丢失的问题，我用手动ACK+本地消息表来兜底。"

这段话里"课程方案"是坦诚，"压测发现"是主动性，"换成Redisson/RabbitMQ"是增量，"手动ACK兜底"是问题解决能力。**面试官想听的从来不是标准答案，而是你做技术决策的过程。**
