## 逐项详情

### 1. Redis BitMap 签到模块

**做了什么：** 新建完整的签到功能链路——`Sign` 实体 → `SignMapper` → `ISignService` → `SignServiceImpl` → `SignController`，提供四个接口：

| 端点 | 功能 | Redis 命令 |
|:---|:---|:---|
| `POST /sign` | 今日签到 | `SETBIT key offset 1` |
| `GET /sign/count` | 本月签到天数 | `BITCOUNT key` |
| `GET /sign/consecutive` | 从今天往回的连续签到天数 | `BITFIELD key GET u{dayOfMonth} 0` |
| `GET /sign/records` | 本月所有签到日期列表 | `BITFIELD key GET u{daysInMonth} 0` |

**Redis Key 设计：** `sign:{userId}:{yyyy:MM}`，每月一个 key，offset = 日期 - 1。

**核心实现要点：**

- **签到判重**：先 `GETBIT` 检查今天是否已签，已签则返回错误，否则 `SETBIT`。
- **BITCOUNT**：直接统计整个 key 中值为 1 的 bit 数，即本月签到总天数。
- **连续签到**：`BITFIELD` 一次性取回第 1 天到今天的 bit 值（`unsigned(dayOfMonth)`），然后从最高位（今天）往低位遍历，遇到 0 停止计数。
- **签到记录**：`BITFIELD` 取回整月 bit 值，逐位判断哪些天是 1。

**为什么用 BitMap 而不是 SET/HASH：**

| 方案 | 单用户单月内存 | 适用场景 |
|:---|:---|:---|
| SET | ~200 bytes（30个member） | 签到数据少，查询灵活 |
| HASH | ~300 bytes（30个field） | 需要存额外签到信息 |
| **BitMap** | **4 bytes（31 bit ≈ 4 bytes）** | **只需记录签到/未签到，极致省内存** |

BitMap 的 31 天只需 4 字节，百万用户签到也只要 4MB，远优于 SET/HASH。

**面试可聊增量：**
> "签到模块我用了 Redis BitMap。每个月一个 key，offset 等于日期减 1，SETBIT 签到、GETBIT 判重、BITCOUNT 统计天数。连续签到用 BITFIELD 一次取回整段 bit 值，从今天往回数遇到 0 就停。一个用户一个月只占 4 字节，百万用户签到只要 4MB。"

可延伸：BitMap 底层是 String 类型的 bit 数组、BITFIELD 支持有符号/无符号整数读写、BitMap 与 HyperLogLog 的适用场景对比。

### 2. Redis SET 关注 / 取关 / 共同关注

**做了什么：** 在已有 `FollowServiceImpl` / `FollowController` 上扩展四个接口：

| 端点 | 功能 | Redis 命令 |
|:---|:---|:---|
| `PUT /follow/{id}/{isFollow}` | 关注 / 取关 | `SADD` / `SREM` + DB insert/delete |
| `GET /follow/or/not/{id}` | 是否已关注 | `SISMEMBER` |
| `GET /follow/common/{id}` | 共同关注列表 | `SINTER`（交集） |
| `GET /follow/count/{id}` | 关注数 | `SCARD`（`opsForSet().size()`） |

**Redis Key 设计：** `follows:{userId}`，SET 中存储被关注用户的 ID。

**核心实现要点：**

- **关注**：`SADD follows:{userId} {followUserId}` + DB insert `tb_follow`，双写保证数据一致。
- **取关**：`SREM` + DB delete，同样双写。
- **防自关注**：`userId.equals(followUserId)` 时直接拒绝。
- **共同关注**：`SINTER follows:{myId} follows:{targetId}` 取两个 SET 的交集，返回 `UserDTO` 列表。

**为什么用 SET 而不是 ZSET/LIST：**

- 关注关系只需要判断"是否存在"和"取交集"，不需要排序和权重。
- SET 的 `SISMEMBER` 是 O(1) 复杂度，`SINTER` 是 O(N*M) 其中 N 是较小集合的大小。
- ZSET 适合需要按时间排序的场景（如 Feed 流），关注关系不需要。

**面试可聊增量：**
> "关注体系用 Redis SET 实现。关注就是 SADD，取关就是 SREM，判断是否关注用 SISMEMBER（O(1)），共同关注用 SINTER 取交集。同时双写 MySQL 做持久化兜底。"

可延伸：SET 的底层实现（intset → hashtable）、SINTER 的时间复杂度优化、大规模用户的关注关系用图数据库（Neo4j）。

### 4. 关注 Feed 滚动分页

**做了什么：** 在 `BlogController` 新增 `GET /blog/of/follow`，在 `IBlogService` / `BlogServiceImpl` 新增 `queryPageOfFollow(Long maxTime, Integer offset)` 返回 `ScrollResult`。

**前端期望的请求/响应格式：**

```
GET /blog/of/follow?offset=0&lastId=1782373751313

Response: { list: [Blog...], offset: 2, minTime: 1782373700000 }
```

**核心算法（滚动分页）：**

```
1. SMEMBERS follows:{userId}     → 获取关注用户 ID 列表
2. WHERE user_id IN (...) AND create_time < maxTime
   ORDER BY create_time DESC     → 取 offset + PAGE_SIZE 条
3. 计算下一页 minTime（末条记录的 createTime 时间戳）
4. 统计末尾同一秒的记录数作为下次 offset（去重同秒数据）
5. subList(offset, size)         → 截取有效记录
```

**为什么用滚动分页而不是传统分页（LIMIT offset, size）：**

| 方案 | 问题 |
|:---|:---|
| `LIMIT 10, 5` | 深度分页性能差（MySQL 扫描 15 行丢弃前 10 行）；用户刷新期间新数据插入导致数据偏移 |
| **滚动分页**（lastId + offset） | 每次从上次最后一条的时间戳继续往前取，不受新数据影响，性能恒定 |

**面试可聊增量：**
> "关注 Feed 用的是滚动分页，不用传统 LIMIT offset。前端传 lastId（上次最后一条的时间戳）和 offset（同一秒跳过的条数），后端查 create_time < lastId 的数据按时间倒序取一页。这样用户刷新期间有新博文插入也不会导致数据重复或遗漏。"

可延伸：Feed 流推模式 vs 拉模式 vs 推拉结合、Timeline 系统的 Redis ZSET 实现、微博 Feed 架构。

```

    → GET /follow/common/undefined（404 + NullPointerException）
    → GET /blog/of/user?id=undefined（404）
```

修复第一个接口后，后续三个请求全部恢复正常。

**面试可聊增量：**
> "查别人信息返回 UserDTO 而不是 User 实体，利用 BeanUtil.copyProperties 只拷贝公开字段（id、nickName、icon、city），手机号和密码不会暴露给前端。"

### 7. Redis SET 点赞系统

**做了什么：** 改造点赞相关的三个端点：

| 端点 | 改造前 | 改造后 |
|:---|:---|:---|
| `PUT /blog/like/{id}` | 无脑 `liked + 1`，重复点赞无限叠加 | Redis SET 切换：已赞→SREM+liked-1，未赞→SADD+liked+1 |
| `GET /blog/likes/{id}` | **不存在（404）** | 查询 Redis SET → 批量取用户信息 → 返回 UserDTO 列表（最多 5 个） |
| `GET /blog/{id}` | 不返回 `isLike` | 新增 `fillIsLike()`：`SISMEMBER` 判断当前用户是否已赞，设置 `blog.isLike` |

**Redis Key 设计：** `blog:liked:{blogId}`，SET 中存储点赞用户的 ID。

**核心实现：**

```java
// 点赞/取消点赞（切换状态）
Boolean isMember = stringRedisTemplate.opsForSet().isMember(key, userId.toString());
if (Boolean.TRUE.equals(isMember)) {
    // 已赞 → 取消：SREM + liked-1
    stringRedisTemplate.opsForSet().remove(key, userId.toString());
    update().setSql("liked = liked - 1").eq("id", blogId).gt("liked", 0).update();
} else {
    // 未赞 → 点赞：SADD + liked+1
    stringRedisTemplate.opsForSet().add(key, userId.toString());
    update().setSql("liked = liked + 1").eq("id", blogId).update();
}
```

**`gt("liked", 0)` 防护：** 取消点赞时 `liked - 1` 加了 `gt("liked", 0)` 条件，防止 Redis 和 DB 数据不一致时 `liked` 变为负数。

**面试可聊增量：**
> "点赞用 Redis SET 实现切换状态——SISMEMBER 判断是否已赞，已赞就 SREM + DB liked-1，未赞就 SADD + DB liked+1。查询博文详情时额外调一次 SISMEMBER 填充 isLike 字段，前端根据这个布尔值控制点赞按钮颜色。查询点赞用户列表直接从 SET 取成员，批量查 DB 返回 UserDTO。"

可延伸：Redis 和 DB 的双写一致性（先写 Redis 还是先写 DB）、点赞数据的异步落库、高并发下的点赞计数精度问题。

### 8. Docker Compose 一键部署

**做了什么：** 新建 `docker-compose.yml` 编排四个中间件服务：

| 服务 | 镜像 | 宿主机端口 | 数据卷 |
|:---|:---|:---|:---|
| MySQL | `mysql:8` | 3307:3306 | `mysql-data`（compose 管理） |
| Redis | `redis:latest` | 6379:6379 | `myredis-data`（external，绑定已有卷） |
| RabbitMQ | `rabbitmq:3-management` | 5672 / 15672 | `rabbitmq-data`（compose 管理） |
| Nginx | `nginx:1.30.2` | 8080:8080 | 挂载 `./nginx/conf/` + `./nginx/html/` |

**配套文件：**

- `nginx/default.conf`：`/api/` 代理到 `upstream backend`（`192.168.100.1:8081/8082`），`/imgs/` 也代理到后端，静态资源走 `/usr/share/nginx/html/hmdp`。
- `sql/` 目录：挂载到 MySQL 的 `/docker-entrypoint-initdb.d/`，首次启动自动执行 `.sql` 初始化脚本。

**踩坑记录：**

| 问题 | 原因 | 解决 |
|:---|:---|:---|
| `unknown variable 'default-authentication-plugin'` | MySQL 8.4 移除了该参数 | 从 command 中删除，默认 `caching_sha2_password` 与 cj 驱动兼容 |
| `WARN: version attribute is obsolete` | Docker Compose V2 不再需要 `version` 字段 | 删除 `version: '3.8'` |
| Redis unhealthy | 旧匿名卷随容器删除被清理，新卷空启动无问题 | 第二次启动正常 |
| compose 拉取了新镜像而非用本地已有的 | compose 文件里的 image tag 与本地不一致 | sed 替换为本地已有 tag |
| `docker rm` 后匿名卷丢失 | 匿名卷随容器删除自动清理 | 改用命名卷（named volume），只有命名卷能跨容器生命周期保留数据 |

**面试可聊增量：**
> "中间件部署用 Docker Compose 编排，MySQL、Redis、RabbitMQ、Nginx 四个服务一键启动。数据卷用命名卷持久化，Redis 绑已有的 external 卷保留历史数据。Nginx 做反向代理，/api/ 转发到后端 upstream，/imgs/ 也代理到后端解决跨机器图片访问问题。"

可延伸：Docker 网络模式（bridge / host / none）、Compose depends_on + healthcheck 的启动顺序控制、多阶段构建 Dockerfile。

## Redis 数据结构速查（面试用）

### 本项目 Redis Key 总览

| Key 格式 | 数据结构 | 用途 | 所在模块 |
|:---|:---|:---|:---|
| `sign:{userId}:{yyyy:MM}` | BitMap | 月度签到 | 签到 |
| `follows:{userId}` | SET | 关注列表 | 关注 |
| `blog:liked:{blogId}` | SET | 点赞用户 | 点赞 |
| `login:token:{token}` | HASH | 登录用户信息 | 登录 |
| `login:code:{phone}` | String | 验证码 | 登录 |
| `cache:shop:{id}` | String（JSON） | 商铺缓存 | 缓存 |
| `lock:shop:{id}` | String | 分布式锁 | 缓存 |
| `seckill:stock:{voucherId}` | String | 秒杀库存 | 秒杀 |
| `shop:geo:{typeId}` | GEO | 商铺地理坐标 | GEO |
| `feed:{userId}` | ZSET | Feed 推模式 | Feed |

### BitMap 常用命令

| 命令 | 功能 | 本项目用法 |
|:---|:---|:---|
| `SETBIT key offset value` | 设置指定位置的值 | 签到：offset = 日期 - 1 |
| `GETBIT key offset` | 获取指定位置的值 | 签到判重 |
| `BITCOUNT key` | 统计值为 1 的 bit 数 | 本月签到天数 |
| `BITFIELD key GET uN offset` | 取回 N 位无符号整数 | 连续签到 / 签到记录 |

### SET 常用命令

| 命令 | 功能 | 本项目用法 |
|:---|:---|:---|
| `SADD key member` | 添加成员 | 关注 / 点赞 |
| `SREM key member` | 移除成员 | 取关 / 取消点赞 |
| `SISMEMBER key member` | 判断是否为成员 | 是否已关注 / 是否已点赞 |
| `SINTER key1 key2` | 取交集 | 共同关注 |
| `SCARD key` | 成员数量 | 关注数 |
| `SMEMBERS key` | 获取所有成员 | 关注列表（Feed 查询） |
